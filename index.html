# Offline vector maps in multiplatform apps   ### Oscar Fonts ### Micho García ### [geomati.co](http://geomati.co) I'm going to tell you the story of a development The resulting by-product has aroused certain interest We are dedicated to *web mapping*, and we have been exploring the benefits of *vector tiles* We have created some interesting toys with *Mapbox GL JS* ![Catastro](ejemplo-catastro.png) ![Turf](ejemplo-turf.gif) ![Trips](ejemplo-trips.gif) ![Editor](ejemplo-editor.gif) So we were asked to develop an App with offline maps and decided to use *Vector Tiles* # Why? # Small size * **Barcelona city** from OpenMapTiles: *12 MB*. * **Half a milion buildings** from cadastre: *37 MB*. * **Catalonia (250x250km)** at scale 1:25 000: *350 MB*. # Flexibility * We can draw data at any *zoom* level and apply any *style*. * We can render those 500.000 constructions in *3D* on a mobile device, symbolize on the fly... even animate them with music! We have an awesome Javasript API: # It's called *Mapbox GL JS* Is there any better alternative? So we are decided to take advantage of the MapboxGL JS API. ![MapboxGL](mapbox-gl.png) If you google for "Mapbox GL JS Offline" ## The official answer is * No, **JS and offline, not**. * Use the **native tween libraries** from Mapbox. * If you want multiplatform, use **MapboxGL React Native**. ## But don't like it * We want our JS API - code reuse, also in desktop browser. * Let's try anyway and see what happens. # First iteration: * Mapbox GL JS as is, inside Cordova. * All online resources are loaded. It works out of the box. Check. # Second iteration: * Move online resources locally (inside `www`). * That is: `sprite`, `glyphs` y `tiles`. Problems. # Problems: * URLs have to be absolute, not relative: ```json "glyphs": "fonts/{fontstack}/{range}.pbf" ``` vs. ```json "glyphs": "file:///www/fonts/{fontstack}/{range}.pbf" ``` * The sprite URLs starting with `file:///` won't load. It's a MapBox GL bug, so: [PR 4649](https://github.com/mapbox/mapbox-gl-js/pull/4649) # And what about tiles? # A lo bruto. Estructura de directorios: ``` 0/0/0.pbf 1/0/0.pbf 1/0/1.pbf 1/1/0.pbf 1/1/1.pbf ``` Au, arreglado. Todos a casa. # No tan rápido * `18/262144/262144.pbf` Miles de directorios, con miles de teselas en cada uno, desparramados en `www`. Eso no hay quien lo gestione. # Tercera iteración: *`.mbtiles`* Un único fichero que contiene todas las teselas. ## El problema ahora es: Cómo leer un *`.mbtiles`* desde Mapbox GL JS # Que se divide en dos subproblemas: * Primero, ¿cómo extraigo una tesela de un *`.mbtiles`*? * Y luego, ¿cómo se la paso a MapboxGL JS? # ¿cómo extraigo una tesela de un *`.mbtiles`*? # ¿Qué es un *`.mbtiles`*? ### Una BDD SQLite con una tabla: "`z`", "`x`", "`y`", "`tesela`". Muy fácil: * Googlear "Cordova SQLite Plugin". * Existen plugins varios. * Este nos sirve: [`cordova-sqlite-ext`](https://github.com/litehelpers/cordova-sqlite-ext). Ya podemos recuperar una tesela `pbf` y la tenemos en un objeto JS. ### La segunda parte del problema: ¿cómo se la pasamos a MapboxGL JS? * La librería sólo sabe pedir teselas a URLs. * Hay quienes han montado un servidor HTTP local embebido en la App. Otros hemos modificado la librería MapboxGL JS. ![Munch](munch.jpg) No es para tanto Reescribir un método "`loadTile`" para que donde dice "AJAX" diga "SQL" Y reempaquetar ![Ciencia](ciencia.jpg) (sí, el infierno está en los detalles) Aprovechamos para encapsular la ñapa de las URLs absolutas (véase segunda iteración) Y convertimos todo eso en una librería reutilizable. ### Apps en Play Store (no son nuestras) * [OpenMapTiles](https://play.google.com/store/apps/details?id=com.klokantech.osm2vectortile) * [Catalunya Offline](https://play.google.com/store/apps/details?id=com.icgc.catoffline) Fin. ¡No tan rápido amigo!
